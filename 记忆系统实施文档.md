# 记忆系统实施文档

## 一、设计原则

1. **单一持久化层**：只用 LangGraph checkpointer 作为所有对话状态的存储，不叠加 SQLChatMessageHistory 或文件系统
2. **SQLite 优先**：摘要、偏好等结构化数据存 SQLite 表，不用 JSONL 或零散文件
3. **按需激活**：LLM 通过工具主动查询历史，不在每轮对话自动注入全部记忆
4. **最小依赖**：第一期只加 1 个依赖（`langgraph-checkpoint-sqlite`），不做向量搜索

## 二、架构

```
┌──────────────────────────────────────────────────┐
│               Agent 调用层                         │
│    create_react_agent(                            │
│        model=model,                               │
│        tools=ALL_TOOLS,                           │
│        checkpointer=SqliteSaver,                  │ ← 持久化
│        prompt=system_prompt + 用户偏好注入          │
│    )                                              │
└──────────┬────────────────────┬───────────────────┘
           │                    │
           ▼                    ▼
┌──────────────────┐  ┌─────────────────────────────┐
│  SQLite DB        │  │  工具层（按需激活）           │
│  memories/        │  │                             │
│  sessions.db      │  │  list_sessions              │
│  ├─ checkpoints   │  │  recall_session             │
│  ├─ writes        │  │  summarize_session           │
│  ├─ session_summaries │                             │
│  └─ preferences   │  └─────────────────────────────┘
└──────────────────┘
```

### 存储层 vs 工具层的关系

- **存储层**：LangGraph checkpointer 在每轮执行后自动持久化消息和图状态，应用层无需手动操作
- **工具层**：LLM 通过 `list_sessions` / `recall_session` 按需回顾历史，不自动注入全部 context
- **偏好注入**：启动时从 `preferences` 表取 top-5 高置信度条目注入 system prompt

## 三、数据库结构

一个 `sessions.db` 文件，包含以下表：

### LangGraph checkpointer 自动创建的表

```
表: checkpoints
功能: 存储图状态快照（含全部消息）
用途: Agent 会话持久化、状态恢复
```

```
表: writes
功能: 存储每步执行的中间结果
用途: 调试、回溯
```

### 应用层自定义表

```sql
CREATE TABLE IF NOT EXISTS session_summaries (
    thread_id TEXT PRIMARY KEY,
    title TEXT,              -- LLM 生成的一句话标题
    summary TEXT,            -- LLM 生成的 150 字总结
    message_count INTEGER,
    created_at TEXT DEFAULT (datetime('now')),
    updated_at TEXT DEFAULT (datetime('now'))
);
```

```sql
CREATE TABLE IF NOT EXISTS preferences (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    key TEXT NOT NULL,
    value TEXT NOT NULL,
    confidence REAL DEFAULT 0.5,   -- 0.0-1.0，随确认次数递增
    session_id TEXT,               -- 触发此偏好的会话
    evidence TEXT,                 -- 触发时的对话片段
    created_at TEXT DEFAULT (datetime('now'))
);
CREATE INDEX IF NOT EXISTS idx_preferences_key ON preferences(key);
```

## 四、核心流程

### 4.1 会话持久化

```
系统启动
    │
    ├─ 打开/创建 memories/sessions.db（SQLite）
    ├─ 创建 SqliteSaver(conn)
    ├─ 创建 Agent(graph, checkpointer=SqliteSaver)
    │
    ├─ 用户发送消息
    │   └─ Agent.run(message, thread_id="abc123")
    │       └─ LangGraph 执行并自动保存状态到 checkpoints 表
    │
    ├─ 用户切换会话
    │   └─ Agent.run(message, thread_id="def456")
    │       └─ 自动加载 def456 的上次状态（如有）
    │
    └─ 下次启动
        └─ Agent(graph, checkpointer=SqliteSaver(同路径))
            └─ 所有历史会话自动恢复
```

每轮对话结束后，LangGraph 的 checkpointer **自动**保存最新状态到 SQLite，不需要手动调用任何保存函数。

### 4.2 历史回顾（按需激活）

```
用户：我们上次聊过 LangChain 记忆对吧？
    │
    ├─ LLM 调 list_sessions()
    │   → 查 session_summaries 表，返回：
    │     [abc123] 2026-06-25 — "LangChain 记忆系统设计" — 5 轮
    │     [def456] 2026-06-24 — "DeepSeek API 配置" — 3 轮
    │
    ├─ LLM 看到 abc123 标题匹配
    │
    ├─ LLM 调 recall_session("abc123")
    │   → 查 checkpointer 的 get_state() 获取完整消息
    │   → LLM 基于历史回答用户
    │
    └─ 用户不再追问
        └─ 不加载更多内容，继续当前对话
```

### 4.3 偏好提取

```
用户：不对，我不是这个意思
    │
    ├─ Agent 检测到不满信号（关键字匹配："不对"、"重做"等）
    │
    ├─ 本轮对话结束时，调 LLM 分析最近 3 轮交互
    │
    ├─ LLM 返回：
    │   [{"key": "response_style", "value": "concise_first"},
    │    {"key": "language", "value": "chinese_priority"}]
    │
    ├─ 写入 preferences 表
    │
    └─ 下次对话开始时：
        ├─ SELECT key, value, MAX(confidence) FROM preferences
        │   GROUP BY key, value ORDER BY confidence DESC LIMIT 5
        └─ 注入 system prompt
```

### 4.4 摘要生成

不每轮生成，仅在以下时机触发：
- 用户显式调 `summarize_session` 工具
- 用户切换到新会话时自动对上一个会话生成一次

摘要内容由 LLM 生成一段话（角色 + 话题 + 结论 + 待办），存入 `session_summaries` 表。`list_sessions` 工具直接查此表。

## 五、工具设计

### 工具一：`list_sessions`

```
功能：列出所有历史会话的标题、时间和摘要
触发条件：LLM 判断需要回顾历史时自动调用
实现：SELECT * FROM session_summaries ORDER BY updated_at DESC
返回格式：
  编号 abc123 — 2026-06-25 14:30 — "LangChain 记忆系统" — 5 轮对话
  编号 def456 — 2026-06-24 09:15 — "DeepSeek API 配置" — 3 轮对话
```

### 工具二：`recall_session`

```
功能：加载指定会话的详细内容
参数：
  - session_id: str (必需)
  - summary_only: bool (默认 True)
行为：
  - summary_only=True: 查 session_summaries 表返回摘要（省 token）
  - summary_only=False: 通过 checkpointer.get_state() 获取完整消息列表
```

### 工具三（可选）：`summarize_session`

```
功能：为指定会话生成或刷新摘要
参数：
  - session_id: str (可选，默认当前会话)
触发：用户显式要求时调用
实现：调 LLM 生成摘要，INSERT OR REPLACE 到 session_summaries
```

## 六、跨会话偏好合并

不使用 LLM 合并，直接 SQL 聚合：

```sql
SELECT key, value, MAX(confidence) as confidence
FROM preferences
WHERE key IN (
    SELECT key FROM preferences
    GROUP BY key
    HAVING COUNT(DISTINCT value) > 1
)
GROUP BY key, value
ORDER BY confidence DESC
LIMIT 5;
```

**为什么不用 LLM 合并？**
- LLM 合并随会话数线性增长 token 消耗
- 偏好本质是键值对，SQL 聚合足够
- 简单、可预测、不依赖模型能力

## 七、system prompt 注入

```python
def build_system_prompt(memory_db_path: str, max_preferences: int = 5) -> str:
    prefs = load_top_preferences(memory_db_path, max_preferences)
    pref_text = "\n".join(
        f"- {p['key']}: {p['value']}" for p in prefs
    ) if prefs else ""
    
    prompt = "你是 clawagent，一个 AI 编程助手。\n\n"
    prompt += "## 工具\n"
    prompt += "你可以使用以下工具：get_current_time, greet, read_file, write_file, run_command, "
    prompt += "list_sessions, recall_session, summarize_session。\n\n"
    
    if pref_text:
        prompt += f"## 用户偏好（基于历史对话）\n{pref_text}\n\n"
    
    return prompt
```

在 `create_agent(settings)` 中传递给 `create_react_agent(prompt=...)`。

## 十、一期不做的事情

| 事项 | 原因 |
|------|------|
| ChromaDB / 向量库 | 依赖重，第一期的核心需求不依赖语义搜索 |
| SQLChatMessageHistory | langchain-community 已废弃，功能与 checkpointer 重叠 |
| temp_memory/ 文件目录 | 文件系统并发不安全，SQLite 管理更优 |
| index.jsonl | JSONL 并发写入会损坏数据 |
| LLM 跨会话偏好合并 | Token 消耗随会话数线性增长，SQL 聚合足够 |
| 每轮对话后自动生成摘要 | 浪费 token，仅在切换会话或用户要求时触发 |
| RAG 自动注入 | 第一期目标是"按需回顾"，二期再考虑"自动推荐" |
